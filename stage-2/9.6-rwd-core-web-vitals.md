# 09.06 Core Web Vitals 日常監控 (報表判讀)

嗨，歡迎來到 MoksaWP 的進階客製化課程！在這個章節中，我們將深入探討一個對使用者體驗 (UX) 與 SEO 至關重要的主題：**網站體驗核心指標 (Core Web Vitals, CWV)**。

學會如何「設定」網站效能外掛只是第一步，真正的挑戰在於如何「持續監控」並「解讀報表」，確保你的網站在真實世界中隨時保持最佳狀態。本章節將教你一套完整的日常監控 SOP，讓你能夠自信地判讀數據，並在問題發生時，知道如何使用我們 Moksa 技術棧中的工具來解決。

---

### 什麼是 Core Web Vitals (網站體驗核心指標)？

Core Web Vitals 是 Google 用來衡量使用者在瀏覽網頁時「真實感受」的一組具體指標。它主要由三個部分組成：

1.  **LCP (Largest Contentful Paint - 最大內容繪製)**:
    *   **衡量指標：** **載入速度**。
    *   **白話解釋：** 使用者點進你的網頁後，畫面上「最大」的圖片或文字區塊需要多久才能完整顯示出來。
    *   **標準：** 理想上應在 **2.5 秒**內。

2.  **INP (Interaction to Next Paint - 互動至下一次繪製)**:
    *   **衡量指標：** **互動性**。
    *   **白話解釋：** 當使用者點擊按鈕、開啟選單或與頁面進行互動時，網站需要多久才能給出「視覺上的回應」。(INP 已於 2024 年 3 月正式取代舊的 FID 指標)
    *   **標準：** 理想上應在 **200 毫秒**內。

3.  **CLS (Cumulative Layout Shift - 累計版面配置位移)**:
    *   **衡量指標：** **視覺穩定性**。
    *   **白話解釋：** 頁面載入時，元素 (如圖片、廣告) 是否會突然出現並導致頁面內容「跳動」，造成使用者點錯位置。
    *   **標準：** 分數越低越好，理想上應 **小於 0.1**。

**Moksa 技巧：** 你不需要死記這些名詞。你只需要知道，這三項指標共同決定了使用者覺得你的網站「**快不快、順不順、穩不穩**」。而 Google 非常重視這些指標，會直接影響你的 SEO 排名。

### 日常監控的兩大核心工具

在日常維護中，我們主要依賴 Google 提供的兩種免費工具來監控 CWV。

*   **Google Search Console (GSC):** 用於「**長期趨勢監控**」，它收集了過去 28 天真實 Chrome 使用者的數據 (稱為「現場資料」Field Data)，告訴你網站的整體健康狀況。
*   **PageSpeed Insights (PSI):** 用於「**即時單頁診斷**」，當你需要立即測試某個特定頁面 (例如剛發佈的文章或活動頁) 的表現時，這就是你的首選工具。

### (實作) 監控 SOP Part 1：使用 Google Search Console 判讀長期報告

這是你每週或每月應該執行的例行檢查。

#### 步驟 1：進入 Core Web Vitals 報告

1.  登入你的 [Google Search Console](https://search.google.com/search-console)。
2.  在左側選單中，找到「**體驗**」區塊。
3.  點擊下方的「**網站體驗核心指標**」。

` `
*截圖：Google Search Console 左側選單，圈選「網站體驗核心指標」。*

#### 步驟 2：判讀整體趨勢圖

你會看到兩份報告：「**行動裝置**」和「**電腦**」。

*   **優先關注「行動裝置」報告**，因為 Google 現在主要以行動版網站作為排名依據。
*   報告會將你網站上的所有網址分為三類：「**良好**」、「**需要改善**」、「**不良**」。
*   **你的目標：** 讓「**良好**」網址的數量持續增加，並將「**不良**」的網址數量降到最低。

` `
*截圖：GSC 的 CWV 報告，顯示「良好」、「需要改善」、「不良」的網址數量趨勢圖。*

#### 步驟 3：深入分析「不良」網址

1.  在圖表下方，你會看到「**為什麼網址未評為良好**」的詳細資料區塊。
2.  這裡會直接點出問題所在，例如「**LCP 問題：超過 2.5 秒 (行動裝置)**」或「**CLS 問題：超過 0.1 (行動裝置)**」。
3.  點擊任一問題項目，例如「**LCP 問題：超過 2.5 秒 (行動裝置)**」。

` `
*截圖：GSC 報告下方顯示具體問題列表，例如 LCP、CLS 問題。*

4.  進入後，你會看到受此問題影響的「**網址範例**」。GSC 不會列出所有網址，但會提供代表性的範例。
5.  這就是你接下來需要動手修復的目標清單！

**Moksa 技巧：** 當你使用 **WP Rocket** 或 **ShortPixel** 進行了重大優化後，可以在這個頁面點擊「**驗證修復作業**」。Google 會重新開始監控這些網址，並在約 28 天後更新報告，讓你知道優化是否成功。

### (實作) 監控 SOP Part 2：使用 PageSpeed Insights 進行即時診斷

當 GSC 告訴你某個頁面有 LCP 問題時，就輪到 PageSpeed Insights (PSI) 上場進行精準打擊。

#### 步驟 1：分析指定網址

1.  前往 [PageSpeed Insights](https://pagespeed.web.dev/)。
2.  將 GSC 報告中有問題的網址範例貼到輸入框中。
3.  點擊「**分析**」。

#### 步驟 2：判讀 PSI 報告

PSI 的報告會給你兩組數據：

*   **發掘實際使用者體驗到的問題 (現場資料 Field Data):** 來自真實 Chrome 使用者的數據，和 GSC 的數據來源相同。如果顯示「這項資料不足」，代表這個頁面的流量還不夠大。
*   **診斷效能問題 (研究室資料 Lab Data):** 這是 Google 在一個標準化環境下「模擬」測試的結果。這是你用來「**除錯 (Debug)**」的主要依據。

` `
*截圖：PSI 報告頂部，顯示整體分數以及 LCP, INP, CLS 三項指標的紅綠燈號。*

#### 步驟 3：找出問題根源與解法

往下捲動到「**診斷效能問題**」區塊，這裡會提供非常具體的建議。以下是幾個常見問題以及對應的 **Moksa 解決方案**：

*   **問題：LCP 過高 (載入慢)**
    *   **PSI 建議：** 「適當調整圖片大小」、「提供新一代格式的圖片」。
    *   **Moksa 解決方案：** 這通常是圖片問題。請確保 **ShortPixel** 外掛已安裝並正確設定。前往「媒體庫」，找到該頁面使用的圖片 (通常是首圖)，確認它是否已被壓縮。如果沒有，手動執行壓縮。
    *   **PSI 建議：** 「Largest Contentful Paint 元素」。
    *   **Moksa 解決方案：** PSI 會直接告訴你哪個元素是 LCP。如果是一張圖片，除了壓縮，你還可以考慮在 **WP Rocket** 的「預先載入」分頁中，設定「**預先載入快取影像**」，將這張圖片的 URL 貼上去，強制它優先載入。

` `
*截圖：WP Rocket 的「預先載入快取影像」設定欄位。*

*   **問題：INP 過高 (互動慢)**
    *   **PSI 建議：** 「避免鏈結過多重要要求」、「減少未使用的 JavaScript」。
    *   **Moksa 解決方案：** 這通常是 JavaScript (JS) 太多或太重造成的。
        1.  前往 **WP Rocket** 的「檔案最佳化」分頁。
        2.  啟用「**延遲載入 JavaScript**」功能。這個功能會延後載入非必要的 JS，直到使用者開始捲動或互動，能大幅改善 INP。這是 **Moksa 必開的王牌功能**。
        3.  檢查是否安裝了過多功能複雜但很少使用的外掛，它們可能是拖慢速度的元兇。

*   **問題：CLS 過高 (頁面跳動)**
    *   **PSI 建議：** 「圖片元素沒有明確的 `width` 和 `height`」。
    *   **Moksa 解決方案：**
        1.  好消息是，新版的 WordPress 和優質佈景主題 (如 **Kadence**) 都會自動為圖片加上寬高屬性。
        2.  如果仍有問題，可以前往 **WP Rocket** 的「媒體」分頁，勾選「**為缺少的圖片尺寸新增屬性**」，這可以作為一道保險。
    *   **PSI 建議：** 「避免使用非合成的動畫」。
    *   **Moksa 解決方案：** 在使用 **Kadence Blocks** 或 **Elementor** 製作頁面時，避免使用會大幅改變版面配置的進場動畫 (例如從左側飛入的區塊)。盡量選擇淡入淡出 (Opacity) 或縮放 (Scale) 這類對效能影響較小的動畫。

### 總結：你的日常監控 SOP

1.  **每週一次：** 登入 GSC，檢查「網站體驗核心指標」報告。
2.  **檢視趨勢：** 關注「不良」網址數量是否有異常增加。
3.  **找出範例：** 如果有「不良」網址，點進去看是 LCP、INP 還是 CLS 問題，並複製有問題的網址範例。
4.  **單點診斷：** 將網址貼到 PageSpeed Insights 進行分析。
5.  **對症下藥：** 根據 PSI 的建議，回到你的 WordPress 後台，利用 **WP Rocket**、**ShortPixel** 等工具進行針對性優化。
6.  **驗證修復：** 優化完成後，回到 GSC 點擊「驗證修復作業」，讓 Google 知道你已經處理了問題。

透過這套結合長期監控 (GSC) 與即時診斷 (PSI) 的流程，你就能像專業的網站維護人員一樣，系統性地管理並提升網站的 Core Web Vitals，為你的訪客提供最流暢的瀏覽體驗，同時也為你的 SEO 排名打下堅實的基礎。
